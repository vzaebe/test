01 · Foundation

Философия и принципы

Семь принципов, на которых построена вся система. Каждое решение в любом компоненте — это применение одного из этих принципов. Если новый компонент не подчиняется им — он не должен попасть в систему.

Core
01

Цвет = смысл

Один цвет всегда означает одно состояние. Синий — «в работе». Зелёный — «согласовано». Красный — «возврат». Никогда не используется цвет «потому что красиво».

Пример: бейдж .bi и индикатор .si со статусом «Проверяется» — оба синие везде в системе.
02

Один компонент — одно назначение

Кнопка .bt.p — одна на всю систему. Не существует «такая же кнопка, но чуть синее». Любая вариация = новый класс с понятным именем.

Пример: размеры .sm, .xs — это не разные кнопки, а варианты одной.
03

Hover везде = синий-светлый

Любой интерактивный элемент при наведении подсвечивается фоном --blue-ll и/или рамкой --yak-blue. Это даёт пользователю мгновенную визуальную обратную связь.

Пример: строка таблицы, чип фильтра, иконка действия — все ведут себя одинаково.
04

Фокус-режим для сложных экранов

Когда на экране несколько панелей, активная подсвечивается синим контуром, а остальные приглушаются (opacity:.42; grayscale(.35)). Так пользователь точно знает, где идёт работа.

Пример: экран 8.2 — список процессов остаётся ярким, остальные панели тускнеют.
05

Плотность важнее декора

Это инструмент для специалистов. Базовый шрифт 13px, поля 32px, бейджи 22px. Никаких лишних теней, градиентов и иконок «для красоты» — только функциональные.

Пример: таблицы — 36px высоты строки, 11.5px шрифт. Без альтернативного «удобного» режима.
06

Композиция, а не наследование

Сложные экраны строятся из тех же простых блоков. Карточка процесса = панель + бейдж + кнопки + поля. Не делаем «специальный экран процесса» — комбинируем.

Пример: экран 8.3 = .proc-list + .process-panel + .bg + .bt.
07

Каждый экран — отдельный frame

Любой экран обёрнут в <article class="scr"> с собственным id вида scr-8-03. Это позволяет извлечь его как самостоятельный модуль и перенести в Figma как отдельный кадр.

Пример: 88 экранов = 88 article-блоков с уникальными ID.

05 · Reference

Справочник статусов

Цвет в системе — это не украшение, а словарь. Один цвет = один смысл во всём модуле. Этот раздел стоит до «Компонентов» специально: сначала договариваемся о значениях цвета, и только потом смотрим, как они применяются в кнопках, бейджах и индикаторах.

10. Таксономия историй

В системе три разных сценария истории. Нельзя ставить кнопку или заголовок «История» без уровня сущности.

История изменений СУДП

Системные изменения документа и атрибутов СУДП: регистрация, статусы СУДП, изменение применяемости, подключение маршрута.

История процесса

Действия пользователей по всему процессу: запуск, согласования, замечания, ответы, экспорт XLSX прямым скачиванием.

История этапа

События внутри выбранного этапа: взятие в работу, замечание, ответ, согласование, откат этапа.

XLSX. Выгрузка истории процесса — direct download из action bar процесса или footer side-sheet истории процесса. Отдельный экран или промежуточная модалка скачивания запрещены.

09 · Process

Правила для разработчиков и AI

Этот раздел читают люди и AI-ассистенты перед тем, как менять любой файл системы. Соблюдение этих правил — обязательно.

Conventions

Когда добавлять новый компонент

  1. Аналогичного нет в текущей библиотеке (поиск по этому файлу — обязательный шаг 1)
  2. Использование ожидается в 2+ местах системы
  3. Компонент описан здесь до применения в экране
  4. Имя класса короткое, на латинице, без кириллицы и пробелов
  5. Все цвета и размеры — через var(--token)

Когда менять токен

  1. Изменение цвета — только при изменении брендбука
  2. Изменение шрифта / шкалы — только через ревью команды дизайна
  3. Любое изменение токена = bump major версии DS
  4. Запрещено добавлять новый токен ради одного экрана — расширяем шкалу только при реальной нужде

Конвенции именования

--yak-*
Брендовые токены ПАО «Компания». Не переопределять в компонентах.
--blue/-l/-ll
Семантический цвет + 2 уровня осветления. Шкала основной → -l (фон средний) → -ll (фон лёгкий).
--sp1..sp10
Шкала отступов 4-base. Использовать только эти значения, любой другой px — отклонение.
.bt + .p / .s / .ok
Базовый класс + модификатор. Не .btn-primary, не .button.
.shell-rail / .shell-rb
Канонические имена shell-навигации в DS. Допустим технический алиас .app-rail / .app-nav внутри .scr при встраивании экрана в каталог, если сохранены те же токены и поведение.
.ws3 + .proc-list / .process-panel
Базовый паттерн Workspace3. Допустим профиль .ws-3 / .ws-panel / .ws-ph / .ws-pb для 3-панельных экранов с focus-mode.
.scr + id="scr-8-03"
Каждый экран. ID = scr-{секция}-{номер с лидирующим нулём}.
.is-focus / .has-focus
Состояние компонента (.is-*) и состояние родителя (.has-*). Не путать.
data-status
Семантический атрибут для машин: queue / active / warn / err. Цвет берётся из CSS-селектора.
